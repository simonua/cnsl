# Security And Privacy Decision

Updated: 2026-06-16

## Analytics Decision

The deployed site uses Google Analytics for purpose-limited aggregate site, visit, session, campaign, and feature usage reporting. Analytics storage is enabled so Google Analytics can store its own first-party analytics identifier for reportable visits and sessions. Advertising storage is denied, and Google signals and advertising personalization are disabled. The app does not send an account identity, app-maintained identifier, or other app-authored data intended to identify or profile a visitor. Measurement is implemented only in the maintained analytics script; the tracker ID remains in shared application configuration. There is no in-app analytics preference. Analytics also requires a fail-closed deployment capability rendered only by the GitHub Pages build job. Ordinary local, editor, performance, and browser-verification builds render analytics disabled, even if local routing presents the production hostname. Automated browsers and pages opened in a VS Code-family embedded browser remain separately ineligible and do not load the Google tag.

Browser-context verification is network-hermetic by default. Playwright blocks service workers and aborts every unmatched external HTTP or HTTPS browser request, including requests to the production site, Google Analytics, Google Tag Manager, Cloudflare Insights, and the National Weather Service. Tests that need a production origin or remote response fulfill it explicitly from local fixtures while the external-request guard remains active. This prevents editor and browser-test traffic from appearing as site usage or delivery telemetry, including traffic attributed to a developer VPN endpoint.

App-authored measurement is limited to aggregate public page-path counts, app version adoption, reviewed fixed campaign labels for app-published inbound links, a fixed count of visits opened from the published flyer QR code, use of visible sharing methods, external-link use without destination details, the category of a settings interaction, annual-data-validated favorite pool and team selections, broad pool/team/meet directory-detail opens, fixed swim meet resource names, and coarse install actions. The flyer QR code uses `utm_source=flyer`, `utm_medium=qr`, and `utm_campaign=2026_pool_season`; the home-page sharing QR code uses the same campaign name with `utm_source=app` and `utm_medium=qr`. Only those complete reviewed tuples are mapped to Google Analytics campaign source, medium, and name fields to compare source effectiveness in aggregate. Those tags are removed locally from the home-page address before page measurement, and the full tagged URL is not transmitted as page location. Favorite-setting measurement reports the selected published pool name or stable team identifier, or the fixed `none` value when cleared. Candidate selections are checked against the current annual directory before publication. Directory measurement reports only whether a pool, team, or meet-date detail was opened, never which item, date, or favorite state. Install measurement reports only fixed interaction stages and never platform, device, browser capability, or prompt metadata. Measurement does not transmit arbitrary query or fragment values, a selected filter, layout, theme, refresh value, location choice, or expanded-state value. Page measurement excludes query strings, fragments, and referrers. External-link measurement does not send the link URL, label, or destination host. Location is requested only by the Pools page when a visitor separately enables distance estimates; granted coordinates are used only in the browser to calculate distances and are never tracked, saved, or sent by the app.

Upgrade measurement reports only a validated prior and current application version joined as `<previous> -> <current>`. When prior use is evident but its version is unavailable, the prior value is the fixed `0`; a clean first use does not publish an upgrade event. The app stores the current application version and a pending validated upgrade path separately from visitor preferences until publication. Neither value identifies a visitor or contains a visitor preference.

Preferences stored by this app do not include analytics state or an app-authored analytics identifier. Google Analytics is configured with analytics storage granted so its first-party analytics identifier can support reportable aggregate visit, session, campaign, and feature-use measurement. Favorite pool and team selections are public directory values, not visitor identifiers, and are included only as event-scoped fields for aggregate comparison. Any future analytics change must remain purpose-limited and must not introduce app-authored visitor identifiers, other stored preference values, contact details, requested coordinates, user-entered text, arbitrary query or fragment values, or unsanitized URLs or referrers. Campaign fields require reviewed, fixed labels on app-published inbound links and regression coverage before use. Enhanced measurement or granular device/location reporting must remain disabled unless separately reviewed against this boundary.

## Browser Policy Decision

GitHub Pages does not provide repository-managed HTTP response headers for this deployment, so the delivered HTML publishes a `Content-Security-Policy` meta policy in the shared layout. The policy is enforced by browsers for page-loaded content and must be checked after any new external integration.

| Directive Area | Allowed Sources | Reason |
| --- | --- | --- |
| Default, base URL, manifest, worker | Same origin only | Site artifact and PWA lifecycle. |
| Scripts | Same origin, inline scripts, `https://www.googletagmanager.com`, `https://static.cloudflareinsights.com` | Site JavaScript, structured metadata, the deployed-site Google tag for Analytics, and dynamic Cloudflare JavaScript loaded by the delivery layer. Cloudflare-injected code cannot use a repository-generated nonce or stable hash, so `script-src 'unsafe-inline'` is required. |
| Styles | Same origin and inline styles | Navigation offset calculation, calendar positioning, and status presentation still use controlled runtime styles. |
| Connections | Same origin, `https://api.weather.gov`, Google Analytics endpoints, `https://cloudflareinsights.com` | Annual/static content, weather safety alerts, purpose-limited usage measurement, and Cloudflare Insights beacon delivery telemetry. |
| Images | Same origin, data URLs, Google Analytics endpoints, `https://www.googletagmanager.com` | Delivered images and Google tag analytics beacon transport. |
| Frames and objects | None | No embedded third-party user experience is required. |

The browser policy permits inline executable scripts because dynamic JavaScript injected by Cloudflare cannot carry a repository-generated nonce or stable hash. No nonce or hash source is included in `script-src`, because modern browsers would otherwise ignore `'unsafe-inline'` and block the Cloudflare injection. This weakens the browser's protection against injected inline JavaScript across the page, so the build continues to reject authored inline executable scripts other than structured JSON-LD. The `script-src 'unsafe-inline'` and `style-src 'unsafe-inline'` permissions are acknowledged integration constraints rather than blanket security claims. They should be removed if Cloudflare's delivery integration and supported UI styling can operate without them.

## Weather Service Location Decision

Weather safety alerts use a fixed Columbia-area point representing the pool area (`39.2014,-76.8610`), not a visitor's current location or ZIP code. The app requests [active alerts](https://api.weather.gov/alerts/active?point=39.2014%2C-76.8610) and [forecast metadata](https://api.weather.gov/points/39.2014,-76.8610) from the National Weather Service for that fixed point. The forecast metadata response supplies the National Weather Service forecast URL used for the near-term forecast check. Optional browser location awareness is a separate Pools-page feature used only to calculate visitor-enabled distance estimates in the browser; even when access is granted, the app never tracks, saves, or sends the visitor's current location.

## Validation

- The artifact verifier requires a valid fail-closed analytics deployment marker and the runtime capability gate; requires analytics storage to remain granted for reportable aggregate visit and session measurement; permits only the reviewed flyer campaign tuple and parameter-free flyer visit count for the locally consumed flyer link; permits validated favorite selections while rejecting other transmitted setting values and direct Google tracking calls outside the analytics module; protects the advertising and navigation-data guardrails; requires the documented Cloudflare inline-script exception; and rejects the removed consent loader or consent setting. The build still rejects authored inline executable scripts before producing the artifact.
- Automated accessibility scans exercise all published routes under the enforced default policy.
- The secure-origin release checklist requires browser-tab and installed-PWA analytics transport checks, payload inspection, GA administration review, an explicit reportability result, and a browser-console CSP review while deployed-site analytics is active.
- Published text in About, FAQ, Settings, and What's New describes the visitor-facing privacy behavior.
