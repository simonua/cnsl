# Security And Privacy Decision

Updated: 2026-07-01

## Analytics Decision

The deployed site uses Google Analytics for purpose-limited aggregate site, visit, session, campaign, and feature usage reporting. Analytics storage is enabled so Google Analytics can store its own first-party analytics identifier for reportable visits and sessions. Advertising storage is denied, and Google signals and advertising personalization are disabled. The app does not send an account identity, app-maintained identifier, or other app-authored data intended to identify or profile a visitor. Measurement is implemented only in the maintained analytics script; the tracker ID remains in shared application configuration. There is no in-app analytics preference. Analytics also requires a fail-closed deployment capability rendered only by the GitHub Pages build job. Ordinary local, editor, performance, and browser-verification builds render analytics disabled, even if local routing presents the production hostname. Automated browsers and pages opened in a VS Code-family embedded browser remain separately ineligible and do not load the Google tag.

Browser-context verification is network-hermetic by default. Playwright blocks service workers and aborts every unmatched external HTTP or HTTPS browser request, including requests to the production site, Google Analytics, Google Tag Manager, Cloudflare Insights, and the National Weather Service. Tests that need a production origin or remote response fulfill it explicitly from local fixtures while the external-request guard remains active. This prevents editor and browser-test traffic from appearing as site usage or delivery telemetry, including traffic attributed to a developer VPN endpoint.

App-authored measurement is limited to aggregate public page-path counts, app version adoption, one coarse app mode per browser session, reviewed fixed campaign labels for app-published inbound links, a fixed count of visits opened from the published flyer QR code, use of visible sharing methods, external-link use without destination details, annual-data-validated public pool identities for pool-specific links, the category of a settings interaction, the allowlisted start-page selection, annual-data-validated favorite pool and team selections, annual-data-validated pool feature filter selections, broad pool/team/meet directory-detail opens, fixed swim meet resource names, and coarse install actions. App mode reports only `webpage` or `installed`; it does not report platform, device, browser, install source, or display capability details. The flyer QR code uses `utm_source=flyer`, `utm_medium=qr`, and `utm_campaign=2026_pool_season`; home-page share links use the same campaign name with `utm_source=app` and one fixed medium matching the authored sharing channel: `qr`, `text`, `email`, or `facebook`. Only those complete reviewed tuples are mapped to Google Analytics campaign source, medium, and name fields to compare source effectiveness in aggregate. Those tags are removed locally from the home-page address before page measurement, and the full tagged URL is not transmitted as page location. Favorite-setting measurement reports the selected published pool name or stable team identifier, or the fixed `none` value when cleared. Pool feature filter measurement reports the sorted selected public feature names joined with `|`, or the fixed `none` value when cleared. Candidate selections are checked against the current annual directory before publication. Pool-specific link measurement reports a stable pool identifier only after validating it against the loaded current annual directory, plus a fixed `pool_page` or `pool_schedule` purpose. Directory measurement reports only whether a pool, team, or meet-date detail was opened, never which item, date, or favorite state. Install measurement reports only fixed interaction stages and never platform, device, browser capability, or prompt metadata. Measurement does not transmit arbitrary query or fragment values, other selected filters, layout, theme, refresh value, location choice, or expanded-state value. Page measurement excludes query strings, fragments, and referrers. External-link measurement does not send the link URL, label, or destination host. Location is requested only by the Pools page when a visitor separately enables distance estimates; granted coordinates are used only in the browser to calculate distances and are never tracked, saved, or sent by the app.

Technical-error measurement reports only the published app version, a standard error name, a fixed runtime or promise-rejection context, a validated same-origin app script path or fixed source category, bounded line and column numbers, and an eight-character fingerprint computed locally from the failure. Raw error messages, rejection values, stack traces, query strings, fragments, and external script URLs are not transmitted. Matching fingerprints are deduplicated, and each page reports at most five distinct errors.

Upgrade measurement reports only a validated prior and current application version joined as `<previous> -> <current>`. When prior use is evident but its version is unavailable, the prior value is the fixed `0`; a clean first use does not publish an upgrade event. The app stores the current application version and a pending validated upgrade path separately from visitor preferences until publication. Neither value identifies a visitor or contains a visitor preference.

Start-page measurement reports only `home`, `pools`, or `teams` after validating the selection against the application-owned allowlist.

Preferences stored by this app remain local behavior inputs and are not copied into analytics user properties or app-authored visitor profiles. Analytics may still use Google's first-party analytics identifier for aggregate page, visit, session, campaign, and feature-use measurement. Pool identities, favorite pool and team selections, and pool feature names are public directory values, not visitor identifiers, and are included only as event-scoped fields for aggregate comparison. Pool-specific external-link events report a stable pool identifier only after validating it against the loaded current annual directory. Any future analytics change must remain purpose-limited and must not introduce app-authored visitor identifiers, stored preference values outside the reviewed start-page, favorite, and pool-feature exceptions, contact details, requested coordinates, user-entered text, arbitrary query or fragment values, or unsanitized URLs or referrers. Campaign fields require reviewed, fixed labels on app-published inbound links and regression coverage before use. Enhanced measurement or granular device/location reporting must remain disabled unless separately reviewed against this boundary.

## Browser Policy Decision

GitHub Pages does not provide repository-managed HTTP response headers for this deployment, so the delivered HTML publishes a `Content-Security-Policy` meta policy in the shared layout. The policy is enforced by browsers for page-loaded content and must be checked after any new external integration.

| Directive Area | Allowed Sources | Reason |
| --- | --- | --- |
| Default, base URL, manifest, worker | Same origin only | Site artifact and PWA lifecycle. |
| Scripts | Same origin, inline scripts, `https://www.googletagmanager.com`, `https://static.cloudflareinsights.com` | Site JavaScript, structured metadata, the deployed-site Google tag for Analytics, and dynamic Cloudflare JavaScript loaded by the delivery layer. Cloudflare-injected code cannot use a repository-generated nonce or stable hash, so `script-src 'unsafe-inline'` is required. |
| Styles | Same origin and the hashed initial canvas block | The exact early light/dark canvas, start-page decision gate, fixed-header paint, and pre-stylesheet header control grouping are authorized by SHA-256. Complete layout and responsive positioning are owned by the site stylesheet; other inline style blocks and attributes are not allowed. |
| Connections | Same origin, `https://api.weather.gov`, Google Analytics collection and Google tag endpoints, `https://cloudflareinsights.com` | Annual/static content, weather safety alerts, purpose-limited usage measurement, and Cloudflare Insights beacon delivery telemetry. |
| Images | Same origin, data URLs, Google Analytics endpoints, `https://www.googletagmanager.com` | Delivered images and Google tag analytics beacon transport. |
| Frames and objects | None | No embedded third-party user experience is required. |

The browser policy permits inline executable scripts because dynamic JavaScript injected by Cloudflare cannot carry a repository-generated nonce or stable hash. No nonce or hash source is included in `script-src`, because modern browsers would otherwise ignore `'unsafe-inline'` and block the Cloudflare injection. This weakens the browser's protection against injected inline JavaScript across the page, so the build continues to reject authored inline executable scripts other than structured JSON-LD and the marked early-theme bootstrap. The build permits exactly one inline style block and verifies that its exact SHA-256 appears in `style-src`. It rejects authored style attributes and browser-script style writes, allowing the policy to omit `style-src-attr` entirely.

The Google Analytics connection allowlist follows Google's current GA4 CSP guidance for `*.google-analytics.com`, `*.analytics.google.com`, and `*.googletagmanager.com`. Advertising endpoints remain excluded because advertising storage, Google signals, and advertising personalization are disabled for this deployment.

Cloudflare Client-side security content security rules are a separate delivery-layer policy. A rule using the `Log` action adds a `Content-Security-Policy-Report-Only` response header, and browsers evaluate it independently from the enforced HTML meta policy. When such a rule is enabled, its Scripts directive must allow `'self'`, `'unsafe-inline'`, `https://www.googletagmanager.com`, and `https://static.cloudflareinsights.com`; any Cloudflare-required `'unsafe-eval'` exception remains limited to that delivery-layer rule. Its Connections directive must allow the same origins documented in the Connections row above. Keep the rule in `Log` mode until every published route is free of unexplained violations, then review both policies before changing it to `Allow`. Updating the HTML meta policy does not update or override this Cloudflare rule.

## Weather Service Location Decision

Weather safety alerts use a fixed Columbia-area point representing the pool area (`39.2014,-76.8610`), not a visitor's current location or ZIP code. The app requests [active alerts](https://api.weather.gov/alerts/active?point=39.2014%2C-76.8610) and [forecast metadata](https://api.weather.gov/points/39.2014,-76.8610) from the National Weather Service for that fixed point. The forecast metadata response supplies the National Weather Service forecast URL used for the near-term forecast check. Optional browser location awareness is a separate Pools-page feature used only to calculate visitor-enabled distance estimates in the browser; even when access is granted, the app never tracks, saves, or sends the visitor's current location.

## Validation

- The artifact verifier requires a valid fail-closed analytics deployment marker and the runtime capability gate; requires analytics storage to remain granted for reportable aggregate visit and session measurement; permits only the reviewed app-share and flyer campaign tuples and a parameter-free flyer visit count for the locally consumed flyer link; permits validated favorite and pool feature selections while rejecting other transmitted setting values and direct Google tracking calls outside the analytics module; protects the advertising and navigation-data guardrails; requires the documented Cloudflare inline-script exception and exact initial-canvas style hash; and rejects the removed consent loader or consent setting. The build rejects other authored inline executable scripts, style blocks, style attributes, and browser-script style writes before producing the artifact.
- Automated accessibility scans exercise all published routes under the enforced default policy.
- The secure-origin release checklist requires browser-tab and installed-PWA analytics transport checks, payload inspection, GA administration review, an explicit reportability result, and a browser-console review of both enforced and report-only CSP violations while deployed-site analytics is active.
- Published text in About, FAQ, Settings, and What's New describes the visitor-facing privacy behavior.
